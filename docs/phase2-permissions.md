# Phase 2 權限規格文件：本機角色（admin / user / guest）

> **用途**：本文件為 Phase 2「本機權限」的單一規格依據（single source of truth），後續設定頁切換（#32）與限制點落地（#33）的實作均以此為準。  
> 本文件獨立於 [`docs/phase2-spec.md`](./phase2-spec.md)（下拉選單規格），避免 PR 互相衝突。

---

## 1. 角色定義

| 角色 | 繁中顯示名稱 | 目的 / 使用情境 |
|------|------------|----------------|
| `admin` | 管理員 | 開發／測試人員使用；擁有所有功能，不受任何限制。**預設角色**，避免測試時被自己鎖住。 |
| `user` | 一般使用者 | 日常記錄使用；可新增、編輯、刪除、搜尋，但**不可匯出資料**（保護資料流向）。 |
| `guest` | 訪客 | 唯讀／示範用途；可搜尋、瀏覽記錄，但**不可刪除**（防止誤操作），也不可新增或編輯。 |

---

## 2. 功能權限對照表

| 功能 | admin（管理員） | user（一般使用者） | guest（訪客） |
|------|:--------------:|:-----------------:|:------------:|
| 新增記錄 | ✅ | ✅ | ❌ |
| 編輯記錄 | ✅ | ✅ | ❌ |
| 刪除記錄 | ✅ | ✅ | ❌ |
| 搜尋記錄 | ✅ | ✅ | ✅ |
| 匯出資料（CSV） | ✅ | ❌ | ❌ |
| 切換角色（設定頁） | ✅ | ✅ | ✅ |

> **限制點最小落地（配合 #33）**：
> - `guest`：不可刪除
> - `user`：不可匯出
> - `admin`：全功能，無任何限制

---

## 3. 被禁止操作的 UI 行為

- 被限制的功能按鈕應改為 **`disabled`（禁用）狀態**。
- disabled 按鈕旁邊需顯示**說明文字**（繁中），讓使用者知道原因。  
  例如：「訪客不可刪除記錄」、「一般使用者不可匯出資料」。
- 若某操作無法改為 disabled（如觸發流程中途才判斷），則改以 **Alert 或 Toast** 方式提示（繁中）。

## 4. 權限儲存方式

### 4.1 儲存位置

- 使用 **React Native AsyncStorage**（`@react-native-async-storage/async-storage`）儲存於本機。
- 重新啟動 App 後，角色設定仍保留（不需重新選擇）。

### 4.2 建議 Storage Key 命名

建議使用 `tbdata.role` 作為儲存角色的 AsyncStorage key。

| Key | 可能值 | 說明 |
|-----|--------|------|
| `tbdata.role` | `"admin"` / `"user"` / `"guest"` | 儲存目前角色字串；讀取不到時以預設值 `"admin"` 代替 |

### 4.3 讀取與寫入時機

| 時機 | 動作 |
|------|------|
| **App 啟動時** | 從 AsyncStorage 讀取 `tbdata.role`；若不存在或值無效，則使用預設值 `"admin"` |
| **設定頁切換角色時** | 立即將新角色字串寫入 AsyncStorage `tbdata.role`，並同步更新 App 的全域狀態 |

---

## 5. 預設角色規則

- **預設角色為 `admin`（管理員）**。
- 理由：Phase 2 先以本機測試為主，預設 `admin` 可確保測試人員不會因為角色限制而無法操作功能，避免測試流程被意外鎖住。
- 切換規則：使用者可在設定頁隨時切換至任意角色，切換後立即生效並儲存。

---

## 6. 角色狀態管理建議

- 建議以 React Context（或全域狀態管理）傳遞目前角色，讓各頁面均可讀取角色判斷是否顯示/禁用功能。
- 切換角色後，全域狀態需**立即更新**，不需重新啟動 App。
